理解 LLM API 中 system、user、assistant 等消息角色如何组织一次请求。这节课放在「大模型应用核心」的「LLM API 开发」模块里，目标是帮你把概念和后续的大模型应用实践接起来。

## 学习目标

学完这一节，你应该能够：

- 用自己的话解释“消息结构”解决什么问题。
- 判断它适合出现在 AI 应用链路的哪个位置。
- 用一个小练习验证自己真的理解，而不是只会复述定义。

## 核心概念

- API 开发要把提示词、消息、状态和输出格式分开管理。
- 模型输出应被程序校验，而不是直接信任。
- 用户体验取决于延迟、错误处理和可恢复性。
- 每一次调用都要考虑成本、日志和降级方案。

## 实践步骤

1. 先用一句话写清楚“消息结构”要解决的问题。
2. 准备一个最小样例，不追求复杂，先让流程跑通。
3. 记录输入、处理过程、输出和你观察到的异常。
4. 把结果和 LLM API 开发 中的其他节点对照，确认它在完整链路中的位置。

## 常见误区

- 只记住名词，没有把它放进真实任务里验证。
- 直接追求复杂方案，跳过最小可运行样例。
- 忽略它和“LLM API 开发”其他知识点之间的依赖关系。
